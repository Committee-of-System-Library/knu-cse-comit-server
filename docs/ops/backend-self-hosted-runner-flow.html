요약

GitHub가 이미지를 만들고, `sidowi`의 runner가 실제 서버를 다시 올립니다.

이 배포 구조의 핵심은 역할 분리입니다. GitHub hosted runner는 검증과 이미지 빌드/GHCR push까지만 담당하고, 마지막 서버 반영은 `sidowi`에 붙어 있는 self-hosted runner가 담당합니다.

Why

배포 책임을 클라우드와 서버로 나누면, 테스트 실패와 실제 운영 반영을 분리할 수 있습니다.

Inspect

workflow → runner → GHCR → compose → env → endpoint 순서로 좁히면 됩니다.

Trigger main push / workflow_dispatch
Deploy Target ghcr.io/.../knu-cse-comit-server:latest
Runner Match self-hosted + backend

구성 요소

운영 기준 전체 흐름은 다섯 층으로 나뉩니다. 아래 카드는 흰색 문서 카드로 두고, 오른쪽 패널은 `Why`와 `Inspect`를 먼저 보여줍니다.

실행 순서

아래 탭은 같은 흐름을 두 가지 관점으로 보여줍니다. 먼저 순차 흐름을 읽고, 그다음 runtime 관점으로 보면 이해가 더 빠릅니다.

1

main push 또는 merge 발생

workflow trigger는 push: branches: [ "main" ] 입니다. feature 브랜치 push만으로는 운영 배포가 일어나지 않습니다.

workflow triggerconfirmed
2

test job 실행

GitHub hosted runner에서 ./gradlew test가 먼저 돌고, 여기서 실패하면 이후 배포 단계는 시작되지 않습니다.

Gate 1ubuntu-latest
3

build-and-push job 실행

이어 ./gradlew clean build 후 Docker 이미지를 build하고, GHCR에 latest와 타임스탬프 태그를 push합니다.

Gate 2GHCR push
4

deploy job이 runner를 찾음

runs-on: [ 'self-hosted', 'backend' ] 를 만족하는 runner가 online이어야만 queue를 빠져나옵니다. runner가 없으면 GHCR push까지만 되고 실제 서버는 바뀌지 않습니다.

label matchqueue 탈출 조건
5

runner가 `sidowi`에서 pull + compose 재기동

여기부터는 GitHub 클라우드가 아니라 `sidowi` 안에서 명령이 실행됩니다. runner는 GHCR 로그인 후 최신 이미지를 pull하고, compose를 다시 올립니다.

docker pullcompose up -d
6

새 컨테이너가 comit.env 기준으로 기동

최종 runtime source of truth는 /opt/docker/env/comit.env 입니다. deploy job은 base compose만 사용하기 때문에, 임시 smoke override는 여기서 다시 사라질 수 있습니다.

confirmedimportant runtime note

GitHub가 맡는 일

코드 checkout, 테스트, jar build, Docker image build, GHCR push까지입니다.

  • test job
  • build-and-push job
  • 아직 서버 접속은 하지 않음

GHCR의 역할

최신 backend 이미지를 저장하는 중간 저장소입니다.

  • latest 기준 pull
  • runner가 참조하는 단일 artifact

Runner가 맡는 일

GitHub deploy job을 `sidowi` 내부 명령으로 바꾸는 실행 주체입니다.

  • 현재 이름: Comit_BE
  • labels: self-hosted, backend
  • 실행 위치: /home/yujihun20251/actions-runner/comit-backend

sidowi가 맡는 일

실제 컨테이너 runtime과 env 파일이 있는 최종 실행 환경입니다.

  • compose: /opt/docker/compose/docker-compose.services.yml
  • env: /opt/docker/env/comit.env
  • container: knu-cse-comit-server

운영자가 봐야 하는 것

배포가 어긋나면 항상 “어디 층에서 멈췄는지”부터 좁혀야 합니다.

  • run 상태
  • runner online 여부
  • pull 성공 여부
  • container env / endpoint 확인

소스 오브 트루스

이 페이지는 코드와 운영 문서를 섞어서 설명합니다. 아래 상자들은 무엇이 어디의 진실인지 구분해 줍니다.

Workflow truth

배포 단계와 job 순서는 workflow 파일이 결정합니다.

  • .github/workflows/deploy.yml
  • test → build-and-push → deploy
  • runs-on: [ 'self-hosted', 'backend' ]

Runtime truth

컨테이너가 어떤 값으로 뜨는지는 서버 env와 compose가 결정합니다.

  • /opt/docker/compose/docker-compose.services.yml
  • /opt/docker/env/comit.env
  • 임시값은 comit.compose.override.yml

Runner truth

현재 runner가 online인지, 어떤 label을 갖는지는 GitHub runner 등록 상태가 결정합니다.

  • name: Comit_BE
  • labels: self-hosted, Linux, X64, backend
  • 현재는 user-level process로 기동

Ops narrative truth

왜 이렇게 운영하는지, 어떤 문제를 겪었는지는 ops 문서가 설명합니다.

Failure Map

실패 지점과 1차 점검 포인트

운영에서 제일 빨리 문제를 좁히려면 실패를 층별로 나눠서 봐야 합니다. 아래 보드는 “어디서 멈췄는지”와 “무엇을 먼저 의심할지”를 같이 보여줍니다.

1 deploy job이 queue에만 남음

배포가 GHCR push 이후로 진행되지 않는 가장 흔한 패턴입니다. 이 단계는 대개 runner 매칭이나 online 상태 문제입니다.

runner offline label mismatch repo에 runner 없음
2 GHCR push는 됐는데 서버가 안 바뀜

artifact는 만들어졌지만 실제 서버 반영 단계가 실패한 경우입니다. runner가 deploy job을 잡았는지와 서버 안에서 pull이 됐는지를 먼저 확인합니다.

runner가 deploy 못 잡음 docker pull 실패 compose 재기동 실패
3 컨테이너는 떴는데 동작이 다름

컨테이너 재기동은 성공했지만, 실제 런타임 값이 기대와 다른 경우입니다. 이 단계는 대개 env precedence나 override 중복 문제입니다.

comit.env 값 누락 override 중복 키 base compose만 사용
4 smoke 테스트가 갑자기 회귀

개인 smoke 테스트나 임시 cross-site 설정은 deploy 이후 기본값으로 돌아가기 쉽습니다. staging 상시값과 임시 override를 분리해서 봐야 합니다.

success/error URL 기본값 복귀 CORS origin 누락 SameSite=None 실험값 소실
