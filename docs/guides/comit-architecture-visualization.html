A. 게시글 신고 API

POST /posts/{postId}/reports 요청이 들어왔을 때의 정상 흐름과 실패 분기입니다.

1

PostController

인증된 @AuthenticatedMember를 받고, 신고 메시지를 request DTO에서 꺼내 ReportService에 전달합니다.

2

ReportService

같은 사용자와 같은 대상 조합이 이미 신고되었는지 먼저 확인합니다. 빠른 중복 차단과 DB unique 보장을 함께 사용합니다.

3

PostService

대상 게시글이 존재하고 soft delete 되지 않았는지 확인합니다. 없으면 POST_NOT_FOUND 입니다.

4

MemberService

신고자를 로컬 회원으로 조회합니다. 이후 Report.create(...) 로 도메인 규칙을 통과한 신고를 생성합니다.

5

ReportRepository + DB

신고를 저장합니다. race condition은 DB unique constraint로 최종 차단하고 duplicate key는 REPORT_ALREADY_EXISTS 로 변환합니다.

!

예외 흐름

중복 신고 → 409 REPORT_ALREADY_EXISTS, 대상 없음 → 404 POST_NOT_FOUND, 공백/500자 초과 → 400 INVALID_REQUEST.

B. 관리자 신고 상태 변경 API

PATCH /admin/reports/{reportId}/status 에서 관리자 권한, 활성 신고 조회, 상태 전이가 어떻게 동작하는지 보여줍니다.

1

AdminReportController

현재 principal이 admin인지 확인합니다. 아니면 즉시 FORBIDDEN 입니다.

2

AdminReportService

soft delete 되지 않은 신고만 조회합니다. 삭제된 신고나 없는 신고는 REPORT_NOT_FOUND 입니다.

3

Report.review(...)

현재 상태가 RECEIVED 인지 확인하고, 새 상태가 REVIEWED, DISMISSED, ACTIONED 중 하나인지 검증합니다.

4

상태 반영

상태를 바꾸고 reviewedAt, reviewedBy 를 기록합니다.

!

예외 흐름

관리자 아님 → 403 FORBIDDEN, 신고 없음 → 404 REPORT_NOT_FOUND, 이미 처리됨 → 409 REPORT_ALREADY_REVIEWED, RECEIVED 재요청 → 400 INVALID_REQUEST.

C. SSO 로그인 흐름

/auth/sso/login → auth server → /auth/sso/callback 으로 이어지는 외부 인증 연결 구조입니다.

1

브라우저 → Comit Backend

/auth/sso/login 으로 들어오면 state를 만들고 auth server 로그인 URL로 redirect 합니다.

2

Auth Server

사용자 인증을 마친 뒤 callback?state=...&token=... 형태로 custom JWT를 돌려줍니다.

3

CustomJwtExternalAuthClient

외부 JWT를 검증하고 ExternalIdentity 로 변환합니다. 이 구현은 port 뒤에 숨겨져 있습니다.

4

MemberService

외부 사용자 정보를 로컬 회원과 연결하거나 최초 생성합니다.

5

Cookie + Filter

인증 cookie를 발급하고, 이후 요청마다 SsoAuthenticationFilter 와 MemberArgumentResolver 가 @AuthenticatedMember 를 주입합니다.

RECEIVED

신고가 막 접수된 상태입니다. 관리자 처리 전의 기본 상태입니다.

REVIEWED

관리자가 확인한 상태입니다. 추가 조치가 없더라도 검토 사실이 남습니다.

DISMISSED

조치 없이 종료된 상태입니다. 신고는 유효하지만 처리 결과가 종료입니다.

ACTIONED

실제 조치가 반영된 상태입니다. 이후 재검토용으로 되돌리지 않습니다.

허용 전이

  • RECEIVED → REVIEWED
  • RECEIVED → DISMISSED
  • RECEIVED → ACTIONED
  • 처리 시 reviewedAt, reviewedBy 가 함께 기록됩니다.

금지 전이

  • RECEIVED → RECEIVED
  • REVIEWED → ACTIONED
  • DISMISSED → REVIEWED
  • ACTIONED → REVIEWED

Soft delete 개념

  • ACTIVE → DELETED 로만 이동합니다.
  • 삭제 후 일반 조회와 관리자 조회 모두 기본적으로 active-only 경계를 따릅니다.
  • deletedAt IS NULL 조건이 repository 조회의 핵심 기준입니다.

회원 계정 상태 (MemberStatus)

ACTIVE

정상 계정 상태입니다. 로그인, 글 작성, 댓글 등 모든 기능을 사용할 수 있습니다.

SUSPENDED

일시 정지 상태입니다. suspendedUntil 이전까지 로그인이 차단됩니다.

BANNED

영구 차단 상태입니다. 기간 제한 없이 로그인이 차단됩니다.

허용 전이

  • ACTIVE → SUSPENDED (기간 지정)
  • ACTIVE → BANNED
  • SUSPENDED → ACTIVE (해제)
  • BANNED → ACTIVE (해제)

적용 범위

  • MemberAuthenticationFilter — Bridge 인증 시 차단
  • SsoAuthenticationFilter — SSO 인증 시 차단 (BusinessException re-throw)
  • 차단 시 403 MEMBER_SUSPENDED / MEMBER_BANNED 응답

게시글 · 댓글 가시성 (hiddenByAdmin)

플래그 전이

  • hiddenByAdmin = false → true (숨김)
  • hiddenByAdmin = true → false (복구)
  • soft delete와 독립적으로 동작합니다.

조회 경계

  • 일반 사용자 쿼리: hiddenByAdmin = false AND deletedAt IS NULL
  • 관리자 쿼리: deletedAt IS NULL 만 적용 (숨김 포함)

왜 controller/api와 구현 컨트롤러를 분리했는가

  • 라우팅과 문서 계약을 인터페이스에 모아 API 산출물을 자동화하기 쉽습니다.
  • 구현 컨트롤러는 얇게 유지되어 테스트와 리뷰 범위가 줄어듭니다.

왜 DTO와 Entity를 분리했는가

  • Request DTO는 운반만, Entity는 도메인 규칙과 상태를 담당하게 역할이 명확해집니다.
  • 외부 계약과 내부 모델을 분리해 API 변화가 엔티티 설계를 직접 흔들지 않게 합니다.

왜 service가 비즈니스 규칙과 트랜잭션 경계를 가지는가

  • 중복 검사, 소유권 검사, 권한 판단, 도메인 간 협력을 유스케이스 단위로 묶기 위함입니다.
  • Controller는 HTTP, Entity는 상태, Service는 실행 흐름이라는 분담이 명확해집니다.

왜 entity가 create/review/delete 메서드를 직접 가지는가

  • 생성 규칙과 상태 전이를 엔티티 내부에 모아 invalid state를 만들기 어렵게 합니다.
  • 예: `Report.review(...)`는 가능한 상태 전이만 허용해야 합니다.

왜 soft delete를 쓰는가

  • 커뮤니티 운영에서 삭제 이력과 관계 보존이 중요하기 때문입니다.
  • 다만 soft delete를 넣는 순간 조회 경계도 active-only 기준으로 같이 바뀌어야 합니다.

왜 외부 인증은 port 뒤에 숨기는가

  • 현재는 custom JWT 구현이지만, 나중에 starter나 다른 구현으로 바꿔도 app 내부 코드를 덜 흔들기 위해서입니다.
  • 즉, 공급자 변경 가능성이 있는 경계를 먼저 추상화한 구조입니다.
