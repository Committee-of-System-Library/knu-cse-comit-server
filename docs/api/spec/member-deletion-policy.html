회원 삭제 정책

1. 목표

  • 관리자 회원 삭제 시 계정은 비활성화하되, 커뮤니티의 게시글·댓글 맥락은 유지한다.
  • 프론트엔드 응답 shape는 바꾸지 않고, 기존 authorNickname 필드만 "탈퇴한 사용자"로 노출한다.
  • 좋아요/방문 기록처럼 집계에 영향을 주는 행위 데이터는 함께 정리해 수치 왜곡을 막는다.

2. 적용 규칙

2.1 회원 계정

  • 회원 삭제는 물리 삭제가 아니라 soft delete로 처리한다.
  • deletedAt을 기록한다.
  • 닉네임, 이름, 전화번호, 프로필 이미지, 학번, 트랙 등 개인정보는 비식별화한다.
  • 삭제된 회원은 로그인/프로필 조회/닉네임 수정 대상에서 제외한다.

2.2 게시글·댓글

  • 삭제된 회원이 작성한 게시글과 댓글은 유지한다.
  • 게시글/댓글 응답의 authorNickname은 "탈퇴한 사용자"로 표시한다.
  • 게시글/댓글 본문은 유지한다. 별도 관리자 삭제 정책은 따로 적용한다.

2.3 좋아요·방문 기록

  • 삭제된 회원의 게시글 좋아요는 제거하고 post.likeCount를 함께 보정한다.
  • 삭제된 회원의 댓글 좋아요는 제거하고 comment.likeCount를 함께 보정한다.
  • 삭제된 회원의 게시글 방문 기록(post_daily_visitor)은 제거한다.

2.4 신고 이력

  • 신고 데이터는 운영 이력으로 유지한다.
  • 삭제된 회원이 신고자 또는 검토자인 경우 관리자 화면 표시 닉네임은 "탈퇴한 사용자"를 사용한다.

3. API/프론트 영향

  • 응답 필드 구조는 바꾸지 않는다.
  • 프론트엔드는 기존 authorNickname 렌더링 로직을 그대로 사용한다.
  • 회원 삭제 후 작성자 표시만 "탈퇴한 사용자"로 바뀐다.

4. 구현 메모

  • 관리자 삭제 API는 AdminMemberService.deleteMember()에서 정책을 오케스트레이션한다.
  • 집계성 데이터 정리는 각 도메인 서비스(PostService, CommentService)로 위임한다.
  • DTO 변환은 Member#getDisplayNickname()를 통해 일관되게 fallback을 적용한다.
